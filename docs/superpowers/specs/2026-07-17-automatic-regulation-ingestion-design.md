# 내부규정 자동 갱신 설계

## 1. 목표

폐쇄망 규정 원본 폴더를 짧은 주기로 검사하여 신규·변경 문서를 자동으로 색인하고 검토 대기로 등록한다. 현재 승인된 규정은 계속 검색·다운로드에 사용하며, 감사팀장이 새 버전을 승인한 뒤에만 지정된 시행일부터 최신본으로 전환한다.

이 설계는 현재 단일 Python 서버 프로토타입에서 바로 실행할 수 있어야 하고, 향후 천안도시공사 내부 서버 또는 단일 Docker 컨테이너에 같은 설정으로 배포할 수 있어야 한다.

## 2. 승인된 업무 흐름

1. 서버가 시작되면 지정된 규정 원본 폴더를 한 번 스캔한다.
2. 이후 설정된 간격마다 같은 폴더를 다시 스캔한다.
3. 파일명과 본문 해시가 기존 버전과 같으면 변경 없음으로 기록한다.
4. 신규·변경 파일은 본문을 추출하고 검색 청크를 생성한 뒤 검토 대기 버전으로 등록한다.
5. 승인 전까지 기존 승인본이 검색 및 다운로드의 현재본으로 유지된다.
6. 감사팀장이 새 버전의 시행일을 확인하고 승인한다.
7. 시행일이 도래하면 새 버전이 현재본이 되고 기존 버전은 이전 버전으로 전환된다.
8. 반려된 버전은 검색 및 다운로드 대상에 포함하지 않는다.

## 3. 접근법

### 3.1 채택: 서버 내 주기적 폴링

서버 프로세스 안에서 별도 백그라운드 스레드가 일정 간격으로 `ingest_local_sources`를 실행한다.

장점:

- 추가 패키지 없이 Python 표준 라이브러리만으로 구현할 수 있다.
- 로컬 디스크, 읽기 전용 Docker 볼륨, 네트워크 공유 폴더에 같은 방식으로 적용할 수 있다.
- 파일 시스템 이벤트가 누락되더라도 다음 스캔에서 변경을 다시 감지한다.
- 기존 수동 색인 및 버전 레지스트리 흐름을 재사용할 수 있다.

제약:

- 스캔마다 대상 파일의 해시를 계산한다.
- 여러 서버 인스턴스가 같은 색인 파일을 동시에 작성하는 구성은 지원하지 않는다.

### 3.2 미채택: 파일 변경 이벤트 감시

파일 이벤트 감시는 반응 속도가 빠르지만 `watchdog` 같은 추가 패키지가 필요하고 네트워크 공유 폴더에서 이벤트 유실 가능성이 있다. 폐쇄망 배포와 오프라인 설치 복잡도를 늘리므로 기본 방식으로 채택하지 않는다.

### 3.3 미채택: 외부 cron 또는 systemd timer

외부 스케줄러는 운영체제에 맞춘 별도 설치가 필요하다. 향후 운영 선택지로 사용할 수 있지만 현재 프로토타입의 자체 실행 기능으로는 채택하지 않는다.

## 4. 구성

자동 갱신은 다음 환경변수로 설정한다.

| 환경변수 | 기본값 | 의미 |
| --- | --- | --- |
| `REG_RAG_AUTO_INGEST` | 비활성 | `1`이면 시작 스캔과 주기 스캔을 활성화한다. |
| `REG_RAG_AUTO_INGEST_INTERVAL_SECONDS` | `60` | 자동 스캔 간격이다. 운영 설정은 최소 10초 이상이어야 한다. |
| `REG_RAG_SOURCE_DIRS` | 기존 탐색 경로 | 규정 원본 폴더다. 여러 경로는 운영체제 경로 구분자로 나눈다. |

`--ingest-local`은 기존과 동일하게 일회성 스캔으로 유지한다. `REG_RAG_AUTO_INGEST=1`이면 최초 스캔을 완료한 다음 HTTP 서버를 열고, 이후 주기 스캔을 백그라운드에서 실행한다.

## 5. 구성요소

### 5.1 자동 갱신 서비스

자동 갱신 서비스는 다음 책임만 가진다.

- 백그라운드 스레드 시작과 종료
- 다음 실행 시각까지 대기
- 자동 스캔 콜백 실행
- 실행 상태와 결과 요약 보관
- 예외 기록 후 다음 주기 계속 실행

서비스 상태는 잠금으로 보호한 스냅샷으로 제공한다.

```json
{
  "enabled": true,
  "running": false,
  "interval_seconds": 60,
  "last_started_at": "2026-07-17T01:00:00Z",
  "last_finished_at": "2026-07-17T01:00:04Z",
  "next_run_at": "2026-07-17T01:01:04Z",
  "run_count": 10,
  "consecutive_failures": 0,
  "last_result": {
    "new_count": 1,
    "changed_count": 0,
    "unchanged_count": 75,
    "error_count": 0,
    "imported_chunks": 28
  },
  "last_error": null
}
```

### 5.2 색인 실행 조정자

자동 스캔과 수동 스캔은 같은 실행 조정자를 사용한다.

- 한 번에 하나의 색인 작업만 실행한다.
- 이미 스캔 중이면 두 번째 실행을 시작하지 않는다.
- 수동 API는 HTTP `409 Conflict`와 현재 자동 갱신 상태를 반환한다.
- 규정 레지스트리와 검색 색인 갱신은 같은 쓰기 잠금 안에서 실행한다.
- 검색과 대시보드 조회는 일관된 상태 스냅샷을 사용한다.

### 5.3 기존 버전 레지스트리

기존 버전 레지스트리 동작을 유지한다.

- 새 해시는 `pending` 버전으로 등록한다.
- 동일 해시는 중복 버전을 만들지 않는다.
- 승인 시 `effective_from`을 확정한다.
- 다음 승인 버전의 시행일 전날을 기존 버전의 `effective_to`로 계산한다.
- 현재 날짜를 기준으로 `scheduled`, `approved`, `superseded` 상태를 갱신한다.

## 6. 데이터 흐름

```text
규정 폴더
  -> 주기 스캔
  -> 파일 해시 비교
  -> 신규/변경 문서 본문 추출
  -> 검색 청크 원자적 저장
  -> 버전 레지스트리 pending 등록
  -> 감사팀장 검토
  -> 승인 및 시행일 확정
  -> 현재본 검색/다운로드 전환
```

자동 스캔이 생성한 `pending` 청크는 색인 파일에 존재해도 승인 버전 허용 목록에 포함되지 않으므로 검색 결과에 노출되지 않는다.

## 7. 운영 상태 표시

`GET /api/dashboard`의 기존 응답에 `auto_ingest` 상태를 추가한다.

운영 화면은 다음 정보를 표시한다.

- 자동 갱신 활성화 여부
- 실행 중 여부
- 스캔 주기
- 최근 시작·완료 시각
- 다음 예정 시각
- 최근 신규·변경·오류 건수
- 최근 오류

기존 `마지막 스캔` 정보와 수동 `현재 폴더 색인` 기능은 유지한다.

## 8. 오류 처리

- 파일 하나의 파싱 실패는 해당 파일 오류로 기록하고 나머지 파일 스캔을 계속한다.
- 전체 스캔 콜백이 예외를 발생시키면 자동 갱신 상태에 오류를 기록한다.
- 오류가 발생해도 스레드를 종료하지 않고 다음 주기에 다시 실행한다.
- 연속 실패 횟수는 성공 시 0으로 초기화한다.
- 서버 종료 시 종료 이벤트를 설정하고 백그라운드 스레드가 정리될 때까지 제한된 시간 동안 기다린다.
- 잘못된 스캔 간격은 서버 시작 단계에서 명확한 오류로 거부한다.

## 9. 안전 경계

- 자동 갱신은 신규·변경 파일을 자동 승인하지 않는다.
- 파일이 원본 폴더에서 삭제돼도 기존 승인본을 자동 삭제하거나 폐지하지 않는다.
- 규정 폐지·철회는 별도 승인 워크플로에서 처리한다.
- 운영 시 규정 원본 폴더는 읽기 전용으로 마운트한다.
- 한 개의 서버 또는 컨테이너만 레지스트리와 색인 파일을 작성한다.
- 다중 인스턴스 운영이 필요해지면 공유 DB와 분산 잠금으로 저장소를 교체한다.

## 10. 테스트 기준

### 자동 갱신 서비스

- 지정된 간격으로 스캔 콜백을 반복 실행한다.
- 중지 요청 후 추가 실행하지 않는다.
- 실행 중 중복 스캔을 시작하지 않는다.
- 예외 발생 후 다음 주기에 다시 실행한다.
- 성공 후 연속 실패 횟수를 초기화한다.
- 상태 스냅샷이 내부 상태를 직접 노출하지 않는다.

### 색인 및 버전 흐름

- 동일 파일을 반복 스캔해도 중복 버전을 만들지 않는다.
- 변경 파일은 검토 대기로 등록한다.
- 승인 전에는 기존 승인본만 검색된다.
- 승인 후에는 새 버전이 지정 시행일부터 검색된다.
- 새 버전 승인 전에는 기존 승인본 다운로드가 유지된다.

### API 및 UI

- 대시보드가 자동 갱신 상태를 반환한다.
- 자동 스캔 중 수동 스캔은 `409`를 반환한다.
- 운영 화면에 활성 상태, 주기, 다음 실행, 최근 결과와 오류가 표시된다.
- 기존 단위·API·정적 UI 테스트가 모두 통과한다.
- Playwright가 있는 환경에서는 데스크톱·모바일 E2E를 실행한다.

## 11. 완료 조건

- `REG_RAG_AUTO_INGEST=1`로 서버를 실행하면 최초 스캔 후 주기 스캔이 지속된다.
- 신규·변경 규정은 자동으로 검토 대기에 들어간다.
- 감사팀장 승인 전에는 기존 최신본 검색·다운로드 동작이 유지된다.
- 승인 후 시행일 기준으로 새 버전이 현재본이 된다.
- 자동 스캔 상태와 오류를 API 및 화면에서 확인할 수 있다.
- 서버 종료 시 백그라운드 작업이 남지 않는다.
- 전체 테스트와 문법 검사가 통과한다.
