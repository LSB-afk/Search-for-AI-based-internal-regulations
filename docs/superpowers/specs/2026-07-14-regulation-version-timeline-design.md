# 규정 개정 시계열 검색·다운로드 보강 설계

## 1. 목표

사용자가 규정명을 검색하고 최신본만 옵션을 해제하면, 현재 기준일의 검색 근거는 그대로 유지하면서 같은 규정의 승인된 개정본 전체를 최신 시행일순으로 확인하고 각 버전의 원본 또는 PDF를 내려받을 수 있게 한다.

## 2. 승인된 사용자 흐름

1. 규정 폴더 스캔 또는 파일 업로드로 새 버전을 검토 대기에 등록한다.
2. 감사팀장 시연 화면에서 시행일을 확인하고 승인한다.
3. 통합 검색에서 최신본만을 켜면 기준일에 유효한 버전만 답변 근거로 사용한다.
4. 최신본만을 끄면 같은 답변 근거와 함께 검색된 규정의 개정 시계열을 표시한다.
5. 시계열은 최신 시행일에서 과거 시행일 순으로 정렬한다.
6. 각 버전은 시행기간, 상태, 변경 유형, 파일명, 원본 다운로드, 원본 PDF 다운로드를 제공한다.

## 3. 접근법 비교

### 접근법 A: 기존 검색 응답에 timelines를 추가

기존 results와 answer는 기준일 검색 의미를 유지하고, include_history가 참일 때만 별도의 timelines 배열을 추가한다. 과거 청크 검색은 규정 식별에만 사용하고, 화면은 규정별 버전 메타데이터를 렌더링한다.

장점:

- 기존 검색 답변과 테스트 계약을 깨지 않는다.
- 같은 버전의 여러 청크가 시계열에 중복되지 않는다.
- 검색과 시계열이 한 응답에서 생성되어 화면 상태가 일치한다.

이 방식을 채택한다.

### 접근법 B: results 자체를 모든 과거 청크로 교체

구현은 단순하지만 한 버전의 여러 조항이 섞이고, 답변 요약이 과거·현재 규정을 함께 인용하게 된다. 기준일 검색 의미를 훼손하므로 채택하지 않는다.

### 접근법 C: 별도 시계열 검색 API 추가

기능 경계는 명확하지만 프론트엔드가 검색 후 두 번째 요청을 해야 하고, 질의·권한·기준일 상태가 어긋날 수 있다. 현재 단일 서버 프로토타입에는 불필요한 표면이므로 채택하지 않는다.

## 4. 데이터 계약

POST /api/search 요청의 기존 include_history 필드를 사용한다.

include_history가 거짓이면:

~~~json
{
  "results": [],
  "timelines": []
}
~~~

include_history가 참이면:

~~~json
{
  "results": [],
  "timelines": [
    {
      "regulation_id": "opaque-id",
      "canonical_title": "18. 인사규정",
      "versions": [
        {
          "version_id": "opaque-version-id",
          "status": "approved",
          "effective_from": "2026-04-13",
          "effective_to": null,
          "change_type": "changed",
          "source_file": "18. 인사규정(개정 2026.4.13.).hwp",
          "download": {
            "source": "/api/download/source?id=opaque-chunk-id",
            "source_pdf": "/api/download/source-pdf?id=opaque-chunk-id"
          }
        }
      ]
    }
  ]
}
~~~

timelines에는 pending, rejected, scan_error 버전을 넣지 않는다. scheduled 버전은 승인된 미래 버전이므로 표시하되 검색 답변 근거에는 시행일 전까지 사용하지 않는다.

## 5. 백엔드 설계

### 규정명 정규화

확장자와 파일명 끝의 bare 날짜뿐 아니라 다음 실제 접미사를 제거한다.

- (제정 2023.7.31.)
- (개정 2026.4.13.)
- (전부개정 2026.4.13.)

유니코드는 먼저 NFC로 정규화한다. 접두 번호는 기관 분류의 일부로 유지한다.

### 시계열 생성

1. 현재 기준일 검색으로 기존 results와 answer를 생성한다.
2. include_history가 참이면 승인·시행예정·이전 버전 청크를 날짜 필터 없이 같은 질의와 역할로 검색한다.
3. 매치된 regulation_id를 수집한다.
4. 레지스트리의 버전을 regulation_id별로 묶고 effective_from 내림차순으로 정렬한다.
5. 버전 chunk_ids 중 현재 색인에 존재하는 첫 청크를 기존 불투명 다운로드 URL로 연결한다.

### 상태 일관성

검색 결과의 version_status, effective_from, effective_to는 청크에 저장된 과거 복사본이 아니라 현재 레지스트리 값을 사용한다.

### 업로드

업로드 파일도 레지스트리 스캔 경로를 거쳐 pending 버전으로 등록한다. 청크는 색인하되 승인 전에는 버전 허용 목록에 없으므로 검색되지 않는다. 원래 파일명을 정규화 입력과 화면 표시용 source_file로 유지한다.

### 초기화

샘플 초기화는 검색 색인, 업로드 파일, 버전 레지스트리와 감사 이벤트를 함께 초기화하여 화면과 검색 상태가 어긋나지 않게 한다.

## 6. 프론트엔드 설계

검색 결과 아래에 개정 시계열 영역을 추가한다. 최신본만이 켜져 있으면 영역을 숨긴다. 최신본만이 꺼져 있고 timelines가 있으면 규정별 카드와 버전 행을 표시한다.

각 버전 행에는 다음을 표시한다.

- 현재본, 시행 예정, 이전 버전 상태
- 시행 시작일과 종료일
- 변경 유형
- 원본 파일명
- 원본 다운로드
- 원본 PDF 다운로드

시계열 링크는 API가 제공한 불투명 URL만 사용하며 source_path는 브라우저에 전달하지 않는다.

## 7. 오류와 보안 경계

- 다운로드 가능한 청크가 없으면 버튼 대신 원본 없음 상태를 표시한다.
- 기존 시연용 역할 필터와 mutation 플래그를 유지한다.
- 이 변경은 실제 AD/LDAP 인증이나 파일 ACL을 추가하지 않는다.
- 검색 및 다운로드 감사 이벤트에서 문서 본문과 질의 전문을 저장하지 않는다.

## 8. 인수 기준

- 실제 파일명 형식의 두 개정본이 하나의 regulation_id로 묶인다.
- 최신본만 검색은 기존 기준일 결과와 답변을 유지한다.
- 개정 이력 포함 검색은 같은 규정의 승인된 모든 버전을 최신순으로 반환한다.
- 시계열의 각 버전 원본과 PDF 링크가 실제 다운로드된다.
- 승인 직후 검색 결과와 시계열의 상태가 최신 상태와 일치한다.
- 업로드 버전은 승인 전 검색되지 않고 승인 후 검색된다.
- 샘플 초기화 후 이전 버전과 감사 이벤트가 남지 않는다.
- 전체 단위/API/UI 테스트와 데스크톱·모바일 브라우저 E2E가 통과한다.
