# 천안도시공사 AI 내부규정 검색 시스템 고도화 설계

## 1. 목적

현재 프로토타입을 천안도시공사의 폐쇄망에서 시연 가능한 내부규정 검색 시스템으로 고도화한다. 사용자는 승인된 최신 규정을 기본으로 검색하고, 기준일을 지정하면 당시 유효했던 규정과 개정 이력을 확인할 수 있어야 한다.

이번 단계의 핵심은 다음 네 가지다.

1. 제품명을 `천안도시공사 AI 내부규정 검색 시스템`으로 통일한다.
2. 웹 화면, 검색 서버, 문서 원본, 색인이 외부 서비스 없이 내부 서버에서 동작하도록 구성한다.
3. 지정 폴더의 규정 변경을 감지하고 승인된 최신본을 시계열로 관리한다.
4. 감사팀장을 최상위로 하는 계층형 권한을 시연용 UI로 제공한다.

## 2. 범위

### 포함

- 폐쇄망 단일 서버 실행 구조
- 지정 폴더 규정 스캔
- 파일 해시 기반 신규·변경·중복 판별
- 규정별 버전 및 시행 기간 관리
- 업데이트 검토·승인 화면
- 최신본 우선 검색과 기준일 검색
- 규정별 개정 타임라인
- 역할별 화면 전환과 잠금 표시
- 검색·다운로드·업데이트 이벤트 감사 기록 시연
- HWP/HWPX/PDF 원본 다운로드

### 제외

- 실제 AD/LDAP/SSO 인증
- 실제 전자결재 연동
- 실제 운영 권한 강제 및 보안 인증
- 외부 LLM 또는 클라우드 API 호출
- 규정 본문을 GitHub 저장소에 업로드하는 배포 방식

권한 UI에는 `권한 시뮬레이션` 표시를 두어 실제 접근 통제로 오해하지 않게 한다. 기존 서버 측 역할 필터는 시연 보조 기능으로 유지할 수 있지만, 운영 보안 통제로 간주하지 않는다.

## 3. 권장 접근법

`폐쇄망 폴더 스캔 + 규정 버전 장부 + 관리자 승인` 방식을 사용한다.

- 단순 파일 교체 방식은 빠르지만 과거 시점 재현과 변경 추적이 불가능하다.
- 전자문서관리시스템 전체 구현은 현재 프로토타입 범위를 초과한다.
- 버전 장부 방식은 현재 Python 단일 서버 구조에 추가할 수 있고 최신본·과거본·승인 상태를 명확히 표현한다.

## 4. 시스템 경계

```text
[폐쇄망 사용자 브라우저]
          |
          v
[내부 웹/검색 서버]
   |      |       |
   |      |       +-- [감사 이벤트 장부]
   |      +---------- [규정 버전/검색 색인]
   +----------------- [읽기 전용 규정 원본 폴더]
```

- 운영 실행에는 인터넷 연결이 필요하지 않다.
- 외부 API, 외부 모델, CDN을 사용하지 않는다.
- 규정 원본과 본문 색인은 내부 디스크에만 저장한다.
- GitHub Pages는 코드·화면 시연용으로만 유지하며 실제 규정 검색 운영 경로로 사용하지 않는다.
- 운영판은 `http://내부서버주소:8765`와 같은 단일 내부 URL로 제공한다.

## 5. 핵심 데이터 모델

### Regulation

- `regulation_id`: 규정의 영속 식별자
- `canonical_title`: 정규화된 규정명
- `category`: 정관·기본, 인사·복무 등 분류
- `owner_department`: 담당 부서
- `latest_approved_version_id`: 현재 최신 승인본

### RegulationVersion

- `version_id`: 버전 식별자
- `regulation_id`: 상위 규정
- `source_path`: 원본 경로
- `source_file`: 파일명
- `content_hash`: SHA-256 해시
- `effective_from`: 시행일
- `effective_to`: 효력 종료일
- `detected_at`: 변경 감지 시각
- `approved_at`: 승인 시각
- `approved_by`: 승인 역할
- `status`: 버전 상태
- `change_type`: 신규, 개정, 삭제 감지
- `chunk_count`: 검색 청크 수

### ScanRun

- `scan_id`
- `started_at`, `finished_at`
- `source_roots`
- `new_count`, `changed_count`, `unchanged_count`, `error_count`
- `status`

### AuditEvent

- `event_id`
- `occurred_at`
- `actor_role`, `actor_name`
- `event_type`
- `target_type`, `target_id`
- `summary`
- `result`

## 6. 버전 상태와 전이

버전 상태는 다음을 사용한다.

- `pending`: 변경 감지 후 검토 대기
- `approved`: 승인되어 검색 가능한 버전
- `superseded`: 더 최신 승인본으로 대체된 이전 버전
- `scheduled`: 시행일이 미래인 승인 버전
- `rejected`: 검토에서 반려된 버전
- `scan_error`: 파싱 또는 메타데이터 판별 실패

상태 전이 규칙:

| 현재 | 다음 | 실행자 | 검증 | 감사 이벤트 | 실패 처리 |
| --- | --- | --- | --- | --- | --- |
| 없음 | pending | 스캐너 | 지원 형식·해시·규정명 | `RegulationVersionDetected` | scan_error |
| pending | approved | 감사팀장 UI | 시행일·규정명 확인 | `RegulationVersionApproved` | pending 유지 |
| pending | scheduled | 감사팀장 UI | 미래 시행일·규정명 확인 | `RegulationVersionScheduled` | pending 유지 |
| pending | rejected | 감사팀장 UI | 반려 사유 입력 | `RegulationVersionRejected` | pending 유지 |
| approved | superseded | 승인 서비스 | 같은 규정의 더 최신본 승인 | `RegulationVersionSuperseded` | 기존본 유지 |
| scheduled | approved | 날짜 판정 서비스 | 시행일 도래 | `ScheduledVersionActivated` | scheduled 유지 |

실제 인증은 구현하지 않으므로 승인 행위는 데모 상태 변경이다. UI와 이벤트에는 이 사실을 명시한다.

## 7. 최신본 판정

1. 동일 규정은 정규화된 규정명과 폴더 분류를 기준으로 묶는다.
2. 같은 파일 해시는 중복 색인하지 않는다.
3. 승인된 버전 중 기준일에 유효한 버전을 후보로 한다.
4. 후보 중 `effective_from`이 가장 최근인 버전을 검색 기본본으로 선택한다.
5. 새 버전 승인 시 직전 버전의 `effective_to`를 새 버전 시행일 전날로 설정한다.
6. 시행일이 같은 복수 버전, 시행일 누락, 규정명 충돌은 자동 승인하지 않고 검토 대기로 남긴다.
7. 기준일이 없으면 오늘 기준 최신 승인본만 검색한다.
8. `이전 버전 포함` 옵션을 켠 경우에만 superseded 버전을 결과에 포함한다.

시행일 추출 우선순위는 사용자 확인 메타데이터, 문서 본문, 파일명, 폴더 기준일 순이다. 어느 경로에서도 판별하지 못하면 시행일 미상으로 표시하고 자동 최신본 전환을 금지한다.

## 8. 시연용 권한 계층

| 역할 | 화면 범위 | 대표 기능 |
| --- | --- | --- |
| 감사팀장 | 전체 | 검색, 업데이트 승인, 이력, 권한, 감사 기록 |
| 감사담당자 | 전체 검색·검토 | 검색, 변경 검토, 이력 열람 |
| 부서장 | 일반·소속 부서 | 검색, 문서함, 소속 규정 열람 |
| 일반직원 | 일반 규정 | 검색, 최신 규정, 일반 문서 다운로드 |

- 상단 사용자 메뉴에서 역할을 즉시 전환한다.
- 메뉴, 지표, 문서 잠금 상태가 역할에 따라 달라진다.
- 제한 문서는 제목과 제한 사유만 표시한다.
- 모든 역할 화면에 `시연용 권한` 배지를 표시한다.
- 실제 운영 전에는 AD/LDAP와 서버 측 권한 검증을 별도 과제로 수행한다.

## 10. 오류 및 실패 처리

- 폴더 접근 실패: 기존 승인본 검색을 계속 제공하고 오류를 운영 현황에 표시한다.
- 파일 파싱 실패: 해당 버전만 scan_error로 격리한다.
- 시행일 불명: 최신본 자동 전환을 막고 수동 검토를 요구한다.
- 동일 시행일 충돌: 두 버전을 pending으로 유지한다.
- 색인 저장 실패: 기존 색인을 원자적으로 유지한다.
- 원본 파일 이동·삭제: 다운로드 불가 상태를 표시하되 메타데이터와 이력은 보존한다.
- 서버 재시작: 저장된 버전 장부와 감사 이벤트에서 상태를 복구한다.

## 13. 인수 기준

- 모든 화면에 변경된 제품명이 표시된다.
- 인터넷 연결 없이 로컬 서버에서 검색·다운로드·업데이트 화면이 동작한다.
- 같은 규정의 두 버전을 시간순으로 볼 수 있다.
- 오늘 기준 검색은 승인된 최신 유효본만 사용한다.
- 과거 기준일 검색은 당시 유효했던 버전을 사용한다.
- 새 파일과 변경 파일이 해시 기준으로 pending에 나타난다.
- 승인 시 최신본이 전환되고 이전본은 superseded로 남는다.
- 네 역할을 전환하면 메뉴와 잠금 표시가 달라진다.
- 권한이 시연용임이 화면에서 명확하다.
- 승인, 검색, 다운로드 이벤트가 감사 기록에 나타난다.
- HWP/HWPX/PDF 원본 다운로드가 유지된다.

- 문서 내 시행일·개정번호 추출 정확도 검증
